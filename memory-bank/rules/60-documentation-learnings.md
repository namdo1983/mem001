# 60 - Documentation and Learnings

## Documentation Standard

Write durable project documentation in concise Markdown. Prefer structure over volume.

Good documentation captures:

- What changed.
- Why it changed.
- How to run or verify it.
- What future agents should know.
- Whether sequential-thinking, Baby Steps, or the 3-retry recovery loop changed the outcome.

## Continuous Improvement

Before completing a multi-step task, decide whether there was a reusable learning.

If yes:

1. Add a dated raw entry to `raw_reflection_log.md`.
2. Move durable, broadly useful items into `consolidated_learnings.md` when they are stable.
3. Prune raw entries after consolidation.

## Changelog

Append `changelog.md` for significant project or harness changes. Keep it factual and dated.
